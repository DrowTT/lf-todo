## 3. 当前状态

### 3.1 已具备的产品能力

截至 2026-04-25，项目已经不是基础待办 Demo，而是一个可长期使用的本地优先桌面待办工具。当前封版目标版本为 `v1.4.0 Stable`，已经落地的核心能力包括：

- 系统视图：`全部 / 暂存区 / 已归档` 已作为待办域内入口落地
- 分类管理：新建、重命名、删除、快捷切换
- 任务管理：创建、编辑、删除、完成、拖拽排序
- 子任务：展开/收起、进度统计、拖拽排序、父子完成状态联动
- 优先级：低 / 中 / 高优先级及对应视觉状态
- 截止时间：日期/时间精度、卡片展示、逾期提醒与桌面通知
- 分类内搜索：当前视图内的即时过滤入口
- 全局搜索：独立搜索弹窗、跨分类搜索、键盘选择、结果跳转
- Quick Add：独立悬浮快速录入窗口，默认落点为系统分类“暂存区”，支持 `#分类名`
- 归档：批量归档、单条归档、归档浏览、单条恢复、批量恢复
- 番茄钟：自由专注 / 任务绑定专注、历史计数、桌面通知
- 快捷键：本地快捷键与全局快捷键、自定义绑定、冲突校验
- 撤销：删除任务、删除子任务、归档后的撤销能力
- 窗口与系统集成：托盘、置顶、窗口位置记忆、开机自启、全局唤起
- 数据能力：JSON 备份导出、覆盖恢复、合并导入、已完成任务自动归档
- 更新能力：electron-updater 集成与设置页状态展示
- 草稿恢复：主输入框、子任务输入框、截止日期与优先级草稿恢复

### 3.2 当前产品判断

项目当前更准确的定位是：

> 一个本地优先、执行导向、交互轻快的桌面待办应用。

它已经具备比较完整的“任务执行”“任务捕获”“搜索找回”“本地备份恢复”链路。对个人使用而言，当前能力已经足够进入 Stable 封版阶段。后续默认进入低维护模式，只优先处理数据安全、启动安装、打包发布、严重 UI 阻断、自动更新阻断和依赖安全升级。

## 4. 架构与工程状态

### 4.1 已形成的稳定架构

当前主干架构方向已经比较清晰：

- Electron `main / preload / renderer` 边界清楚
- `shared/contracts` + `shared/types` 已承担共享协议层职责
- renderer 已形成 `runtime + repository + store + facade` 分层
- 数据库已使用 version-based migration，而不是早期的字段探测式升级
- preload / IPC 关键边界已有请求与响应校验
- `useAppBootstrap` 已接管启动流程，避免初始化逻辑散落在叶子组件
- settings / updater 已状态化，而非简单的“点按钮调 API”
- 常见写操作已接入统一的异步动作与 pending 状态模型

### 4.2 已解决的历史问题

此前分散在多份评审文档中的不少问题已经完成收口，当前不再应被视为待办：

- `is_completed` 已统一为 boolean 语义
- `preload` 与共享类型已切到共享模型，而不是继续手工复制
- 任务与分类能力已通过 repository 抽象，不再由 store 直接依赖 `window.api`
- `ConfirmDialog` 已改为组件内部聚焦与键盘处理，不再依赖全局 document 监听
- 主窗口与 Quick Add 窗口都已启用 `sandbox: true`
- `Tray` 已使用模块级引用，避免局部变量 GC 风险
- `AppUserModelId` 已使用实际应用 ID，而不是默认值
- 子任务排序已统一回收到 subtask store，而不是由组件直接持久化
- 会话状态恢复已提前到首次渲染之前，草稿恢复可靠性明显更好
- `pnpm lint:errors` 当前已经通过

### 4.3 仍存在的工程缺口

当前最大的工程短板已经不再是基础架构，而是测试兜底不足：

- 仓库当前没有正式的自动化测试文件与测试脚本
- 关键路径仍主要依赖类型检查、lint 和手工验证
- 系统视图、Quick Add、搜索、归档恢复等高频链路缺少自动化回归保护

## 6. 推荐推进顺序

### 6.0 v1.4.0 Stable 后的默认策略

`v1.4.0 Stable` 后不再默认推进大功能。以下方向仅作为未来可能性保留，不作为当前开发承诺：云同步、移动端、账号系统、AI 整理、看板、甘特图、复杂日历、多主题、插件系统、复杂报表、周期任务和标签系统。

如确需继续开发，应先重新评估产品边界，而不是直接进入实现。

如果后续继续演进，建议按下面顺序推进，而不是并行铺开：

### Phase 1：先补闭环与语义层

- 系统视图模型收敛
- 任务备注 / 描述
- 全局搜索主入口打磨
- 智能视图
- 最小自动化测试基线

目标：

- 让“默认落点”和“系统视图”语义更清楚
- 让用户能更快跨分类找回来
- 让任务具备稳定的上下文承载

### Phase 2：再补长期留存能力

- 外部迁移导入 / 更多导入格式
- 周期任务
- 标签系统
- 回收站
- 批量操作
- 分类图标 / 颜色
- 分类拖拽排序

目标：

- 让组织方式从“单一分类列表”升级为“多视角 + 多属性任务管理”

### Phase 3：最后做差异化体验

- 统计看板 / 图表
- 成就页
- 更丰富的 Quick Add 语法
- 更多导出格式

目标：

- 强化“更快、更轻、更专注”的产品识别度

### 6.1 当前建议先投的三件事

如果只从 CEO / 架构视角挑未来一轮最该投的事项，建议优先固定为下面三项：

1. 系统视图语义收敛
   原因：`全部 / 暂存区 / 已归档` 已有雏形，但模型边界还不够稳，继续叠功能会放大信息架构混乱。

2. 任务备注 `description`
   原因：当前任务仍是短标题模型，复杂事项没有稳定上下文承载区，会限制产品往执行系统升级。

3. 自动化测试基线
   原因：当前主链已经足够长，没有测试兜底，后续每一轮迭代都会被回归成本拖慢。

一句话结论：

> 先把语义层、数据层、验证层补齐，再继续扩视图和长期能力；否则功能越多，结构越乱。

### 6.2 建议延后但保留的迭代点

下面这些点不是不重要，而是建议建立在前三项完成之后再投入：

- 首批智能视图：`今天 / 即将到期 / 全部未完成 / 高优先级 / 最近创建`
- 外部迁移导入
- 周期任务 / 重复任务
- 标签系统
- 回收站 / 软删除
- 批量操作

## 7. 专项结论

### 7.1 搜索设计结论

原搜索专题文档的核心原则仍然成立，但需要按当前实现校正表述：

- 新增待办输入框继续保持为唯一常驻主输入框
- 分类内搜索继续保留为页内轻过滤，不破坏当前上下文
- 全局搜索已经独立为弹窗入口，负责跨分类搜索、结果选择与任务定位
- 当前默认快捷键为 `Ctrl/Cmd + P`；如后续把搜索进一步升格为统一命令面板，再评估是否切到 `Ctrl/Cmd + K`

一句话总结：

> 新增负责创建，分类内搜索负责筛当前列表，全局搜索负责跨分类找回与跳转。

### 7.2 大列表性能结论

原大列表指南建议继续保留：

- 单分类顶级任务超过 `300` 时，再评估虚拟列表
- 不要过早为了性能引入第二套任务数据源
- 如果拖拽排序与内联编辑冲突明显，再考虑显式区分“排序模式”和“编辑模式”

### 7.3 Quick Add 结论

Quick Add 基础能力已经落地，且默认会落到系统分类“暂存区”，但后续仍有两个明确升级方向：

- 继续收敛 `暂存区 / Inbox / 全部` 的命名与文案语义，降低理解成本
- 扩展轻量语法，逐步支持更强的“先记后整理”能力

## 8. 构建、发布与手动验证

### 8.1 本地开发与构建

```bash
pnpm dev
pnpm verify:agent:standard
pnpm build:win
```

Windows 打包目前沿用 `electron-builder`：

```bash
pnpm build:win
```

如需直接发布到 GitHub Release：

```bash
npx -y electron-builder --win --publish always
```

### 8.2 发布注意事项

发布前建议按以下顺序执行：

1. 确认工作区干净
2. 更新 `package.json` 的版本号
3. 完成提交并推送分支与 tag
4. 执行 `pnpm build:win`
5. 如需自动发布，确保环境变量 `GH_TOKEN` 可用
6. 发布后确认 Release 为 Published，并包含核心附件

重点检查项：

- `latest.yml`
- `*-setup.exe`
- `*.blockmap`

### 8.3 手动验证建议

当前仓库没有正式自动化测试，因此保留以下最小手动验证建议：

1. 优先执行 `pnpm verify:agent:standard`
2. 启动应用后验证核心路径：
   - 分类增删改查
   - 任务增删改查与拖拽排序
   - 子任务联动
   - 全局搜索与结果跳转
   - 截止时间与提醒
   - Quick Add
   - 归档与恢复
   - 番茄钟
   - 数据导出、覆盖恢复、合并导入
3. 如需定位数据库问题，可在 DevTools 中调用 `window.api.db.*` 做临时检查

说明：

- 手动数据库测试更适合作为调试手段，而不是长期文档中心
- 后续一旦补上自动化测试，本节应进一步收缩
